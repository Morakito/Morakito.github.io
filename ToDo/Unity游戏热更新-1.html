Unity游戏热更新-1

什么是热更新

无热更新游戏

1，没有热更新的游戏在更新游戏资源（贴图，脚本等）的时候，会让用户去应用市场重新下载完整版的游戏安装包。
2，更新过程太过繁琐，要到游戏外部下载安装包；而且下载的安装包容量可能很大。可能仅仅是更新几个皮肤，就要用户重新下载游戏。因此热更新还是很重要的。

有热更新游戏

1，游戏在启动之后，自动进行更新。（类似于王者荣耀，英雄联盟等的更新）
2，更新流程：

（1）玩家下载游戏
（2）制作后续的游戏更新
（3）将资源打包上传到更新服务器
（4）玩家在启动游戏的时候自动检测是否有更新
（5）自动下载资源包
（6）执行热更（因资源类型而异）

游戏热更新的种类

1，资源热更新
2，代码热更新：平台操作系统的限制，可能将病毒藏在热更新中。

热更新与资源管理

可热更的资源类型

预制体 Prefab
材质 Material 贴图 Texture 模型 Model
场景 Scene
着色器 Shader
动画 Animation
动画机 Animator
Unity中除代码资源外的所有资源。

打包方式

1，按照资源（Label）打包（X）
2，按照资源路径打包
3，编写打包插件
（1）压缩选项：

不压缩
LZ4：基于Block的压缩（将资源分块）
LZMA：基于Chunk的压缩（读取资源的头部信息Header，找到Chunk，再找到Block）

（2）编写版本文件

将版本文件上传
版本号（客户端的版本号和服务器端的版本号对比）

上传

1，上传到HTTP或FTP服务器
2，开发时：
（1）配置一台开发用的资源服务器
（2）可以和开发机在同一台机器上

下载

1，客户端，服务器各有一份版本文件
2，客户端启动时，先下载一份服务器的版本文件
3，用服务器版本与客户端版本做对比
4，判断游戏资源是否改变：md5编码校验

版本比对

要比对的点：

1，增加项（直接下载）
2，减少项（直接删除）
3，更新项（和服务器同步）

比对方式

1，自己实现对比算法
2，调用源码管理工具获取结果

优化

1，缓存版本比对结果（降低服务器负荷）

解包的依赖关系处理

1，Unity在打包的时候会自动记录资源依赖的关系（资源清单文件）
2，我们在解包的时候需要：

读取依赖关系
先加载所有的依赖项，最后加载资源主体

（解包 ！= 加载资源 ！= 实例化对象）

资源引用计数

1，场景中每一个实例化对象会将其引用+1
2，当引用计数为0时应当释放资源

代码热更新

（代码热更新是资源热更新的一个重要特例，将代码以数据的情况隐藏起来欺骗操作系统）

ILRunTime：C#
ToLua/XLua：Lua（祖传代码）

Addressable

1，Addressable不是一个新东西，是对AssetBundle的高级包装。支持：

资源引用计数
自动资源寻址
远程/本地热更新对开发者透明

代码热更新

总体流程

1，编写热更新代码
2，打包为二进制资源包（Lua）
3，和资源更新一样上传，下载，版本对比
4，加载更新后的脚本资源（虚拟机）

代码热更新能做什么

1，配置数据配置（游戏数据） (策划表Excel -> Lua/C#)
2，逻辑代码热更新

为什么代码能热更新

1，虚拟机(Lua虚拟机/.net)
2，方式：反射 / 代码绑定

代码热更新插件的功能

提供了Unity和c#脚本的双向绑定

1，XLua：补丁式热更
2，ToLua：经典热更
3，ILRuntime：反射/CRL绑定热更

代码插件不提供什么

1，热更代码框架

Lua代码热更新框架要实现的功能

1，一系列方便、可用的工具库
2，一个供Unity端调用的基于事件的Lua框架库
3，对Lua代码进行保护（外挂，作弊）

为什么大家都用Lua而不是ILRuntime呢？

1，ILRuntime相对于Lua来说还是个新东西，Lua更加成熟
2，很多公司都已经有一套成熟的Lua热更新框架了，没有动力使用ILRuntime
（Lua热更新是WOW带起来的）
